# Wear OS to iOS BLE Review

## Bottom Line

The right direction is possible, but the current app is a prototype, not a complete bridge. A Wear OS app and an iOS app can communicate over BLE if one side acts as a GATT peripheral/server and the other acts as a GATT central/client. For real bidirectional traffic, both sides also need a message protocol, chunking, acknowledgments, reconnect handling, and native platform channels.

## Important Platform Reality

- iOS can use Core Bluetooth as central and peripheral, but background behavior is constrained. Advertising and scanning in the background are throttled and service discovery behavior changes.
- iOS apps cannot read arbitrary system notifications from other iPhone apps for forwarding to Wear OS. You can forward notifications generated by this app, or data the user explicitly grants through supported APIs such as HealthKit.
- Wear OS can run BLE GATT server/peripheral code, but long-lived advertising should be managed as a foreground service with correct runtime permissions on modern Android.
- `flutter_blue_plus` is mainly the central/client side of BLE. The native Android/iOS peripheral code in this repo is separate and currently not connected to Flutter message streams.
- BLE is not a socket. Single JSON messages can exceed the MTU. The app needs chunking, sequence numbers, checksums, and reassembly before reliable user-facing messaging.

## Main Gaps To Build Next

1. Decide the primary BLE role model.
   - Recommended first version: Wear OS advertises as peripheral/server, iOS scans and connects as central/client.
   - Later version: add the reverse direction only if you need iOS-originated advertising while the watch scans.

2. Connect native BLE peripheral code to Flutter.
   - Android: expose `startServer`, `stopServer`, `sendNotify`, and incoming writes via platform channels.
   - iOS: expose `startPeripheral`, `stopPeripheral`, `sendNotify`, subscription state, and incoming writes via platform channels.

3. Add message framing.
   - Use a binary or compact JSON envelope with `messageId`, `chunkIndex`, `chunkCount`, `payloadLength`, and `checksum`.
   - Reassemble chunks before calling `BridgeMessage.fromJson`.
   - Reject oversized messages early.

4. Add security.
   - Pair with an in-app PIN or QR code.
   - Derive a session key and encrypt message payloads before writing to BLE.
   - Do not rely only on BLE link-layer security for health or personal data.

5. Add background behavior intentionally.
   - Android/Wear OS: foreground service and reconnect strategy.
   - iOS: enable only the required Bluetooth background modes and design around throttling.
   - Avoid promising always-on background sync on iOS until tested on real devices.

6. Limit product claims.
   - Notification sync should mean Bridge app notifications unless you later add a legitimate source.
   - Health sync requires HealthKit on iOS and Health Services / Health Connect style APIs on Android, plus user permission screens.
   - Media control requires platform media APIs, not only a BLE message type.

## Suggested Implementation Milestones

1. Build a minimal ping/pong path:
   - Wear OS advertises.
   - iOS scans, connects, writes `ping`.
   - Wear OS receives native write and notifies Flutter.
   - Flutter returns `pong` through native notify.

2. Add reliable transport:
   - Chunking.
   - Ack timeout.
   - Retry with max attempts.
   - Duplicate detection.

3. Add a pairing screen:
   - Show a 6-digit code on one device.
   - Confirm code on the other.
   - Store trusted peer metadata locally.

4. Add real feature modules:
   - App-owned notifications.
   - Manual health sample transfer.
   - Simple media command demo.

5. Test on physical devices:
   - iPhone with screen on/off.
   - Wear OS watch with screen on/off.
   - Reconnect after Bluetooth toggle.
   - Reconnect after app restart.
   - Payloads larger than MTU.

## Documentation Corrections

The existing Copilot-generated docs are useful as a sketch, but they overstate completion. Treat these phrases as inaccurate until the gaps above are implemented and tested:

- "fully functional"
- "production-ready"
- "complete and ready to use"
- "notification sync" for arbitrary iOS notifications
- "health data sync" without HealthKit and Android health API integration
- "media controls" without native media session integrations
